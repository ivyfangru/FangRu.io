124. Binary Tree Maximum Path Sum
Given a non-empty binary tree, find the maximum path sum.

For this problem, a path is defined as any sequence of nodes from some starting node to any node in the tree along the parent-child connections. The path must contain at least one node and does not need to go through the root.

题目分析：
这道题是给一个非空的二叉树，找到最大的路径和。路径当中最少有一个节点，且不一定必须经过根节点。

Algorithm

4. Median of Two Sorted Arrays

There are two sorted arrays nums1 and nums2 of size m and n respectively.

Find the median of the two sorted arrays. The overall run time complexity should be O(log (m+n)).

You may assume nums1 and nums2 cannot be both empty.

Example 1:

nums1 = [1, 3]
nums2 = [2]

The median is 2.0
Example 2:

nums1 = [1, 2]
nums2 = [3, 4]

The median is (2 + 3)/2 = 2.5

首先，这两个数组是排序的，最初的方法是把这两个数组merge了，然后直接在merge后的数组里取中位数，这个方法的事件复杂度是O(n), 空间复杂度是O(n);

进一步优化的话，因为都是排序数组，因此可以考虑二分。假设一个数组长度是a，另一个是b，那么我们要找的中位数一定在一个数组和一个数组的前半部之间，这个相当于我们的第一次二分，我们要去掉更大的那个1/4。于是我们每一次选aStart + k/2位置的数aMid与bStart + k/2位置的数bMid，对比这两个中位数，如果aMid > bMid, 那么大于aMid的数中一定不可能有我们想要的中位数，那么可以舍去，反之同理。

由于有一种可能是中位数是两数的平均数，所以我们要获得两个中位数，保证我们选取的中位数是正确的。

An introduction to Progressive Web Apps

这篇文章介绍了什么是PWA，它的兴起的原因，好处以及主要应用技术。

好处有

  1. PWA更加轻量。传统手机App至少要200MB，PWA确实在KB级别。
  2. 不需要根据手机系统而编程。
  3. 访问或者获得的成本比较低（针对于用户来说，不需要下载App）
  4. 发布和更新的cost更低了。
  5. 对于深链接支持度比传统App-store app要好。

核心概念

  1. Responsive，UI能够适应各种设备界面
  2. App-like feel，更加像一个application而不是一个网页。
  3. Offline support，使用设备容量来储存offline的使用。
  4. Installable，设备浏览器会提示用户安装app
  5. Re-engaging，一旦安装App弹出提示指示用户使用App
  6. Discoverable，搜索引擎和搜索引擎优化能够带来的用户量远多于App store所带来的用户量。
  7. Fresh，一旦上线，app就能够自我更新和内容更新。
  8. Safe，使用HTTPS
  9. Prograssive，可以在任何设备上访问，甚至是很旧的设备，即使可能会少一些features
  10. Linkable，通过URLs可以很容易链接。

146. LRU Cache
Design and implement a data structure for Least Recently Used (LRU) cache. It should support the following operations: get and put.

get(key) - Get the value (will always be positive) of the key if the key exists in the cache, otherwise return -1.
put(key, value) - Set or insert the value if the key is not already present. When the cache reached its capacity, it should invalidate the least recently used item before inserting a new item.

Follow up:
Could you do both operations in O(1) time complexity?

Example:

LRUCache cache = new LRUCache( 2 /
capacity / );

cache.put(1, 1);
cache.put(2, 2);
cache.get(1); // returns 1
cache.put(3, 3); // evicts key 2
cache.get(2); // returns -1 (not found)
cache.put(4, 4); // evicts key 1
cache.get(1); // returns -1 (not found)
cache.get(3); // returns 3
cache.get(4); // returns 4

这个题的关键是亮点，一个是put和get操作，这个我们只能用HashMap来存Key-Value pair。 另外一个是我们有capacity，要及时vanish不常访问的数据，同时要把最近访问的数据移到前面，那么可以使用的数据结构就是double LinkedList。

这道题的原理其实很简单，用hashmap来存key和对应的node，用一个counter来管理是否超过内存容量。最麻烦的可能就是对链表的修改，可以用两个dummy variable head和tail来指向开始和末尾，head后面跟的是most recent visited，tail前面所指是latest visited。当插入新的元素时，直接加到head后面，如果容量超过capacity就删除tail前面的。

Algorithm

Leetcode 42. Trapping Rain Water

Given n non-negative integers representing an elevation map where the width of each bar is 1, compute how much water it is able to trap after raining.

The above elevation map is represented by array [0,1,0,2,1,0,1,3,2,1,2,1]. In this case, 6 units of rain water (blue section) are being trapped. Thanks Marcos for contributing this image.

Example:
Input: [0,1,0,2,1,0,1,3,2,1,2,1]
Output: 6

首先暴力解法是从左到右遍历时，在第i位置时分别向左向右遍历，找到不小于当前高度的左边最大值left_max和右边最大值right_max，用当中最小值来减去当前高度，就是第i位置的蓄水量。time complexity是o(n ^ 2)，space complexity是o(1)。

接下来，试图降低时间复杂度。从暴力解法中，我们要找到左边最大值和右边最大值，那么，我们可以从左向右遍历，从右向左遍历找到当前位置高度左边最大和右边最大，保存在数组left_max[]和right_max[]当中。这样我们最后从左遍历一遍用不小于当前高度的min(left_max[i], right_max[i]) - height[i]来得到当前蓄水量。这样我们的time complexity是o(n), 但是，space complexity是o(n)。

最后看了discussion，双指针做法有o(n)的时间复杂度和o(1)的空间复杂度。原理是我们在找左边和右边的最大高度，用一个指针left从左向右走，一个指针right从右向左走，同时记录left指针遍历过的左边最大leftH和right指针遍历过的右边最大rightH。如果leftH < rightH, 就用当前left指针所指高度与leftH对比，如果leftH > height[left], 当前蓄水量是leftH - height[left]，否则就是0，并且把leftH替换成height[left]。
同理如果leftH >= rightH，就用当前right指针所指高度与rightH对比，如果rightH > height[right], 当前蓄水量是rightH - height[right]，否则就是0，并且把rightH替换成height[right]。 可以把leftH和rigthH想象成桶壁，我们要做的是选举桶壁，同时记录最大蓄水量。如果当前高度小于最矮桶壁，那么可以蓄水，如果出现新的高度高于前桶壁，那么就被选举成了新的桶壁。

Looking inside the technology that powers Pinterest
这篇文章介绍了Pinterest发展为一个拥有超过2.5亿用户的技术发展理念。

上面写道他们的发展路程是不断的重建系统，有的时候甚至是紧急重建。所以他们精确设计的技术实现是建立在多次的危机和历史问题上的。

去年，Pinterest明确并且编写了他们技术发展理念，总结下来有四个核心：Mission(使命)，Context(内容)，Strategy Principle(战略原则)，Technical Strategies(技术战略)。

  • Mission
    Pinterest技术策略的使命是能够匹配公司的使命：帮助人们发现他们所热爱的东西并且去实现。不会为了发展技术而去发展，相反，而是在支持Pinterest的使命基础上有目的地去发展。
  • Context
    用来衡量Pinterest的产业工程三个衡量变量：复杂的产品(Complex product)，资源(Resources)和野心(Ambition):
  1. Complex product
    Pinterest针对不同的呈现页面有不同的实现方式，比如主页图片展示与图片搜索的算法不同，主页图片要更加个性化，而消费图片和广告图片也有不同的表现模式。普通图片和推广图片，二者在语料库的大小，周期和用户的交互模式也有所不同。
  2. Resources
    作为拥有被成百上千万人使用的产品的成长中的公司，在Pinterest的每一个工程组能够获取更大的资源库。人员分配对于领导团队也是最重要的决定之一，因为总有很多新增的项目可以为用户，合作者以及Pinterest自身提高服务的质量。
  3. Ambition
    在增加全球用户的同时，Pinterest的系统必须转向长期成长的轨道而不能局限在小的渐进的提高。
  • Strategy Principle
  1. Simplicity & Velocity ()
    为了结合我们的工程原则，所有的项目都是从简单的开始然后迭代。我们的技术策略对于工程师团队来说是保证我们有一个清晰的，审慎的，同时又是灵活的指路明灯的基础。这个提供了对的方向的同时给予了我们足够的自由去采用最新的技术。我们建立的所有都需要支持快速的迭代。
  2. Scale():
    我们为了使Pinterest扩大同时为未来数十亿用户和数万亿的图片量作准备而建立系统。我们的技术战略为了保证长期的考量同时抢占先机同时为未来的扩张而建立，而不是实现可能带来昂贵的重建的短期决定而建立的。
  3. Ownership():
    通过保持我们的战略方向而不是严格地规定，我们鼓励工程师拥有平衡速度和质量的决定权。同时，策略本身是在公司内所有组里的工程师建议的基础上建立的。
  4. Resusability(复用原则):
    当专注在正在处理的问题时，我们希望能够找到满足需求的其他的系统。我们团结起来一起建立 几乎一直 跨越内部界限，甚至跨越公司界限使用或者为开源项目作出贡献。我们认真思考如何把使用案例带给更多人使用。我们多方面思考复用同时在速度之间找到最佳平衡点
  5. Focused complexity():
    在我们的领域我们是专业的，需要通晓Pinterest之外的技术。我们能够学习并选择争取的尖端技术并使用在我们选择复杂解决方案的关键领域。我们在非常慎重并且深入地理解利弊后我们接受复杂的方案。
  • Technical Strategies
    我们把这些理念呈现在下面每一个技术策略当中。为了避免不必要的对工程组的破坏我们通过来自技术主管合适的投入来发展这些技术。本着交互理念的精神，技术策略是有生命力的，也将会随着时间进化。

下面是我们现在相对成熟或者还在探索中的技术策略：

  1. Machine Learning(机器学习)：
    机器学习是Pinterest里许多组的关键技术。机器学习能够帮助我们最大化模型试验的速度。我们发展了单一训练模型，同时为我们根本业务和广告案例提供动力。根据这个策略发展的技术减少了建立新的以机器学习为基础的生产应用程序的障碍。

  2. Content Distribution Infrastructure(内容分布式基础架构):
    Pinterest大部分关键use case不论是原生内容还是付费内容都是基于用户的输入。这些实例的技术挑战是相通的，像低时延，支持大规模，避免系统分裂，等等。因此基于模块和使用模式的交集我们建立了允许复用和个性化的相结合地使用策略。其中，这些模块包括了增量更新的倒排索引，键值储存，分散集中层，和图片遍历架构等。

  3. Data Management(数据管理):
    Pinterest有强大的数据驱动的通过现实实践的决策的文化。为了保证我们工程和技术决策的正确，我们加强和维护关键业务数据的可靠性，提高程序员的生产力，并且提高数据管道的投资回报率。数据管理策略专注于数据管制、质量、发现和编码模式。

  4. Data Processing(数据处理):
    构建了庞大的数据管理模式后，这个模式就给特定的logging，查询处理，软件开发框架和数据处理系统一个方向。
    Data Processing: Building off the broader data management strategy above, this track provides strategic direction on specific logging, query processing, programming frameworks and data processing systems.

  5. Experimentation(实验):
    我们迭代速度基于我们实验的速度。在实验性策略指导下，把我们实验基础架构和方法的演化设置为10倍的实践吞吐量。

  6. Cloud(云服务):
    云计划最初的目的是记录在2到4年的战略方法。清晰的基础架构方向将保证Pinterest保持高可用性，高恢复力，高性能，高使用率，高成本效率和可预测的。

  7. Supported Languages(支持语言):
    在Pinterest，我们使用多种编程语言，包括Java，Python，JavaScript，C++和Go。在我们工程师团队对于每一种语言都有一定程度支持。编程语言策略提供了一个能够快速，安全和低消耗地开发我们的产品的语言分析和建立语言支持的框架。

  8. Core Client Platform(核心客户端平台):
    这个技术规划了一个建立客户端技术可以产生一个快速注册用户的模式，与平台模式相结合，利用传统设备兼容性并且根据用户模式，网络情况和服务器响应的情况来快速响应。

  9. API(API模式):
    API模式所有的API和Pinterest的API端口提供了连贯和一致的方向。包括为被用户使用的产品功能提供服务的内部API和给合作方提供的外部API，第三方应用程序开发者和第三方产品集成的外部API。

Geohash
学习了Geohash算法，这个算法根据经纬度把地址转化成一串由数字和字母组成的字符串。当获得两个地址，不需要计算两个地址的直线距离，只要从把两个用Geohash转化来的字符串做对比，前面相同的字符越多，距离越近，大大减少了计算时间。
